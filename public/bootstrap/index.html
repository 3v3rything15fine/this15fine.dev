Pre-install Checklist

Before you begin, make sure you have everything ready. Think of this as gathering your ingredients before cooking.

You will need:

  • The Framework Desktop, assembled with the Noctua fan installed
  • 4TB NVMe drive installed in the back-side M.2 slot (cooler position)
  • 2TB NVMe drive installed in the slot near the heatsink
  • Two Samsung T7 external drives connected via USB-C (do NOT format these)
  • Two YubiKey security keys (a primary and a backup). If you only have one, buy a second before starting.
  • A USB flash drive with Fedora Workstation 43 on it
  • A phone or laptop with internet access (for verifying login URLs)
  • A pen and paper (you will write down important recovery codes)
  • A fireproof envelope or safe (for storing recovery codes)
What are YubiKeys? They are small USB security keys. When plugged in, they let the computer unlock your encrypted drives without typing a long password. You tap the key when it blinks. Think of it like a physical house key for your computer.

Check the NordVPN account. Open https://account.nordvpn.com in a browser and make sure you can log in. You need an active subscription.

Check Tailscale. Open https://login.tailscale.com and make sure you can log in. This is the secure remote-access tool.

Verify the Fedora USB. Plug the USB flash drive into the Framework Desktop. You will boot from it later by pressing F12 (or ESC) at startup and selecting the USB drive.

Quick home inventory. Check that the Hue bridge is version 2 (it should say so on the bottom). No action needed, just good to know.

Decide where to store recovery codes. You will get two long codes (64 characters each). Plan to store them in: (1) the fireproof envelope and (2) the TXST password manager.

Install Fedora + Encrypt the Drives
30 min work + 30 min wait ▼
What happens in this phase You will install the operating system (Fedora) and set up encryption so nobody can read the drives without your YubiKey or recovery code. This is the longest phase, but most of the time is waiting for things to install.
Part A: Installing Fedora

Boot from the USB drive. Power on the Framework Desktop. Immediately press F12 (or ESC) repeatedly until a boot menu appears. Select the USB drive from the list.

Start the installer. Fedora will load a live desktop. Look for "Install to Hard Drive" (or "Install Fedora") on the desktop or in the Activities menu. Click it.

Language and keyboard. Pick English (or your preferred language). Click Continue.

Installation Destination. This is the important part. Click "Installation Destination." You will see the drives listed. Select the 4TB drive only (it is the primary). Then click "Custom" at the bottom (not "Automatic"). Click Done.

Set up partitions. You need to create these partitions on the 4TB drive. Use the "+" button to add each one:

Mount PointSizeTypeNotes
/boot/efi1 GiBEFI SystemLeave as default
/boot2 GiBext4Unencrypted boot files
/60 GiBbtrfs (LUKS encrypted)Check "Encrypt" box. Set a strong passphrase.
/home300 GiBbtrfs (same LUKS)Personal files live here
/var150 GiBbtrfs (same LUKS)Logs and container data
/dataRemaining spacebtrfs (same LUKS)Everything else: media, AI models, cloud storage
Important When it asks for an encryption passphrase, pick something strong and memorable. This is a backup password in case both YubiKeys are lost. Write it down temporarily.

Accept the layout. Click "Done" at the top. A summary will appear. If anything looks wrong, go back. If it looks right, click "Accept Changes."

Begin Installation. Click the big blue button. Wait for it to finish. This takes about 15-20 minutes. Create your user account when prompted (username: jasonobrien).

Reboot. When the installer says "Complete," click Reboot. Remove the USB drive when the screen goes dark. The machine will restart and ask for your encryption passphrase. Type it to unlock, then log in.

Part B: Post-Install Tuning
Opening a terminal Press the Super key (the key between Ctrl and Alt), type "Terminal," and click the Terminal app. All the commands below go in this window.

Turn off compression on /data. This tells the system not to compress files stored in /data, since media files and AI models are already compressed.

sudo btrfs property set /data compression none

Set size limits on each area. This prevents one area from filling up and starving the others. Like giving each area its own shelf in the fridge.

sudo btrfs quota enable --simple-quotas /
sudo btrfs qgroup limit 60g /
sudo btrfs qgroup limit 300g /home
sudo btrfs qgroup limit 150g /var
sudo btrfs qgroup limit 3400g /data

Enable regular drive maintenance. This keeps the SSD drives healthy by cleaning up unused space periodically.

sudo systemctl enable --now fstrim.timer

Set up snapshot recovery tools. Snapshots let you "rewind" the system to an earlier state if something goes wrong. Like a save point in a video game.

sudo dnf install grub-btrfs timeshift
sudo systemctl enable --now grub-btrfsd.service
sudo timeshift --create --tags D --comments "post-install baseline"

Verify the system did not set up sleep mode. This machine needs to stay awake at all times (it runs services). This just checks that the safety net is working.

systemctl status systemd-oomd.service

You should see "active (running)" in green text.

Part C: Setting Up YubiKey Encryption
Read this entire section before starting This section involves plugging in and unplugging YubiKeys in a specific order. Getting the order wrong is not dangerous but will mean you have to redo steps. Have both YubiKeys and a pen/paper ready.

Find the encrypted drive names. This command shows which drives are encrypted. You need the device names for the next steps.

lsblk -f | grep -i luks

You should see two entries. The 4TB drive is probably nvme0n1p3 and the 2TB drive is probably nvme1n1p1. To confirm:

sudo cryptsetup luksDump /dev/nvme0n1p3 | grep -E '(Version|UUID)'

It should say "Version: 2" -- that means LUKS2 is working.

Generate recovery keys. These are your emergency backup codes. If both YubiKeys are lost or broken, these codes are the ONLY way to unlock your drives. Each command will print a long string of characters. Write them down exactly, character by character.

sudo systemd-cryptenroll --recovery-key /dev/nvme0n1p3

Write down the code and label it "4TB recovery key."

sudo systemd-cryptenroll --recovery-key /dev/nvme1n1p1

Write down the code and label it "2TB recovery key."

Do not skip this Store these recovery keys in TWO places: (1) a fireproof envelope and (2) the TXST password manager. If you lose both YubiKeys and do not have these codes, the data on the drives is gone forever. There is no backdoor.
STOP. Unplug the BACKUP YubiKey. Leave ONLY the PRIMARY YubiKey plugged in. Label it "Primary."

Enroll the primary YubiKey. This teaches the drives to accept this specific key. When the key blinks, tap it.

sudo systemd-cryptenroll --fido2-device=auto /dev/nvme0n1p3
sudo systemd-cryptenroll --fido2-device=auto /dev/nvme1n1p1
STOP. Unplug the PRIMARY YubiKey. Plug in ONLY the BACKUP YubiKey. Label it "Backup."

Enroll the backup YubiKey. Same process, different key.

sudo systemd-cryptenroll --fido2-device=auto /dev/nvme0n1p3
sudo systemd-cryptenroll --fido2-device=auto /dev/nvme1n1p1

Edit the encryption config file. This tells the system to look for the YubiKey at boot instead of asking for a password every time. The command opens a text editor.

sudo cp /etc/crypttab /etc/crypttab.bak
sudo nano /etc/crypttab

For each line in the file, find the last section (after the word "none") and add: discard,fido2-device=auto

A line should look something like:

luks-abcd-1234  UUID=abcd-1234  none  discard,fido2-device=auto

Save the file: press Ctrl+O, then Enter, then Ctrl+X to exit.

Rebuild the boot image. This updates the startup process to include YubiKey support. It takes a minute.

sudo dracut -f --regenerate-all

Verify the YubiKey software is in the boot image.

sudo lsinitrd /boot/initramfs-$(uname -r).img | grep -iE 'libfido2|fido2'

You should see something like libfido2.so.1 in the output. If you see nothing at all, run these two commands:

echo 'install_items+=" /usr/lib64/libfido2.so.1 "' | sudo tee /etc/dracut.conf.d/fido2.conf
sudo dracut -f --regenerate-all

Then run the verification command again.

Back up the encryption headers. If the drive's header gets corrupted, this backup can restore access. Like backing up the lock, not just the key.

sudo cryptsetup luksHeaderBackup /dev/nvme0n1p3 --header-backup-file /tmp/luks-4tb.img
sudo cryptsetup luksHeaderBackup /dev/nvme1n1p1 --header-backup-file /tmp/luks-2tb.img
sudo chown jasonobrien:jasonobrien /tmp/luks-*.img
mv /tmp/luks-*.img ~/
chmod 600 ~/luks-*.img

Copy these files to: the laptop, a T7 drive, and a fireproof USB.

Part D: Test Everything Works
This part involves multiple reboots You will restart the computer three times to verify that each way of unlocking works. Have the recovery key you wrote down ready.
REBOOT 1: Power off. Remove ALL YubiKeys. Plug in ONLY the PRIMARY YubiKey. Power on.

Test primary YubiKey. The machine should boot and the YubiKey should blink. Tap it. The system should unlock and reach the login screen without typing any password.

REBOOT 2: Power off. Remove ALL YubiKeys. Plug in ONLY the BACKUP YubiKey. Power on.

Test backup YubiKey. Same thing. Tap when it blinks. Should unlock just like the primary.

REBOOT 3: Power off. Remove ALL YubiKeys (none plugged in). Power on.

Test recovery key. Without a YubiKey, the system will show a password prompt. Type the 64-character recovery key exactly as you wrote it down. It should unlock.

If any test fails Do not panic. You can always type the original encryption passphrase you set during installation. Go back to Part C and re-check the steps.
✓ After Phase 1: Drives are encrypted. YubiKeys are enrolled. Recovery keys are saved. Machine boots with a YubiKey tap. Safe to stop here and pick up tomorrow.

VPN Setup
45 min ▼
What is the VPN for? The VPN (NordVPN) is used only for the download containers. It keeps download traffic private. The rest of the computer uses the normal home internet.

Create a special system user for downloads. This creates a dedicated account that the download software runs under. You will need the number it prints later.

sudo useradd --system --no-create-home --shell /sbin/nologin mediavpn
MEDIAVPN_UID=$(id -u mediavpn)
echo "mediavpn UID: $MEDIAVPN_UID"

Write down the UID number. You will need it in the next step and in Phase 4.

Create the helper scripts directory.

mkdir -p ~/bin
NordVPN Config Extraction
This part is a bit involved NordVPN does not give you a simple config file. You have to install their app, connect once to grab the secret settings, then remove the app. It sounds odd but it is the official way.

Install the NordVPN app (temporarily).

sudo dnf install https://repo.nordvpn.com/deb/nordvpn/centos/nordvpn-release-1.0.0-1.noarch.rpm || \
  sh <(curl -sSf https://downloads.nordcdn.com/apps/linux/install.sh)

Start NordVPN and add yourself to its group.

sudo systemctl enable --now nordvpnd
sudo usermod -aG nordvpn $USER
newgrp nordvpn

Log in to NordVPN.

nordvpn login
Switch: Terminal to Browser
You will see a URL in the terminal. Open it in your browser and log in with the NordVPN account. After logging in, come back to the terminal.

Connect to a US server using the fast protocol.

nordvpn set technology nordlynx
nordvpn connect United_States

Extract the secret config. This grabs the connection settings that NordVPN created and saves them to a file the system can use directly.

sudo wg showconf nordlynx > /etc/wireguard/nordlynx-us.conf
sudo chmod 600 /etc/wireguard/nordlynx-us.conf
What just happened You captured the encryption keys and server address that NordVPN uses internally. Now the system can connect directly without the NordVPN app.

Disconnect and remove NordVPN. You do not need the app anymore. The config file has everything.

nordvpn disconnect
nordvpn logout
sudo dnf remove nordvpn

Test the direct connection. This connects using the extracted config. The curl command checks your public IP address. It should show a different city/state than your home.

sudo wg-quick up nordlynx-us
curl ipinfo.io

If the IP is NOT your home IP, it is working. Now turn it off for the remaining setup:

sudo wg-quick down nordlynx-us
Killswitch Setup
What is a killswitch? If the VPN connection drops, the killswitch blocks the download software from using the normal internet. It prevents accidental unprotected downloads. Like a circuit breaker.

Create the killswitch rules file. Replace <MEDIAVPN_UID> with the UID number you wrote down earlier.

sudo mkdir -p /etc/nftables
sudo tee /etc/nftables/vpn-killswitch.nft <<EOF
table inet vpn_killswitch {
  chain output {
    type filter hook output priority filter; policy accept;
    meta skuid <MEDIAVPN_UID> oifname != "nordlynx-us" drop
  }
}
EOF
Replace the placeholder Where it says <MEDIAVPN_UID>, type the number from the first step (like 987 or whatever it was). The finished line should look like meta skuid 987 oifname != "nordlynx-us" drop

Set up the killswitch to load at boot.

sudo systemctl daemon-reload
sudo systemctl enable vpn-killswitch.service

Set up password-free access for VPN commands. This lets the VPN scripts run without asking for a password every time.

sudo visudo -f /etc/sudoers.d/texmf-wrappers

In the editor that opens, paste this single line:

jasonobrien ALL=(ALL) NOPASSWD: /usr/bin/wg-quick, /usr/sbin/nft, /usr/bin/systemctl reload firewalld, /usr/bin/systemctl restart nftables, /usr/sbin/cryptsetup, /usr/bin/mount, /usr/bin/umount, /usr/sbin/losetup, /usr/sbin/wg

Save and exit.

✓ After Phase 3: vpn on works. Download containers will be protected once they are running. curl ipinfo.io shows a NordVPN IP when VPN is on, your home IP when off.

Create the Private Research Vault
20 min ▼
Think of this as a locked filing cabinet inside an already-locked office The whole computer is already encrypted (that was Phase 1). This creates a second encrypted space INSIDE the computer specifically for sensitive research data. It has its own separate password and only opens when Jason is actively working with research files. When locked, even the rest of the computer cannot read what is inside.

Create the vault file. This makes a 100GB container file. It does not actually use 100GB of space right away -- it grows as data is added.

sudo truncate -s 100G /data/research.img
sudo chown jasonobrien:jasonobrien /data/research.img
chmod 600 /data/research.img
sudo chattr +C /data/research.img

Encrypt the vault. When it asks for a passphrase, use a DIFFERENT password than the one from Phase 1. This is the second layer of protection.

Important Use a DIFFERENT passphrase than the one you set during Fedora installation. This separation is the point -- the research vault has its own key. 
sudo cryptsetup luksFormat --type luks2 /data/research.img

Type "YES" (in all caps) when it asks to confirm. Then enter your new passphrase twice.

Open, format, and set up the vault folders.

sudo cryptsetup luksOpen /data/research.img research
sudo mkfs.ext4 /dev/mapper/research
sudo mkdir -p /mnt/research
sudo mount /dev/mapper/research /mnt/research
sudo mkdir /mnt/research/{inbox,working,outbox}
sudo chown -R jasonobrien:jasonobrien /mnt/research
chmod 700 /mnt/research

Test locking and unlocking. Lock it up:

sudo umount /mnt/research
sudo cryptsetup close research

Now verify it is locked -- this command should fail:

ls /mnt/research/
ls: cannot access '/mnt/research/': No such file or directory

That error means it is working. The vault is locked and invisible.

Day-to-day usage Jason uses research unlock to open the vault (it asks for the passphrase) and research lock to close it. Simple as opening and closing a safe. The golden rule: when in doubt, lock.
✓ After Phase 7: The research vault is ready. research unlock opens it, research lock seals it. Ready for IRB-approved data whenever that arrives.
