Morning routine

Daily briefing fires at 7:00 AM via cron. Merges Google Calendar + Outlook into a single view. Pushes to ntfy so it lands on the phone before coffee kicks in.

The briefing includes: today's events, pending tasks, carry-forwards from yesterday's daily note, and anything Claude flagged overnight.

Work sessions

Claude Code sessions with persistent memory, a skill system (reusable prompt templates), and the ability to spawn parallel agents for multi-track work.

A typical session: Jason opens a terminal, types claude, and says what needs doing. Claude reads memory files, checks the task board, picks up context from last session. Work happens. Files get written. Commits get made. The daily note gets updated.

The key insight: Claude isn't a tool you use, it's a collaborator you work with. The memory system makes that real instead of metaphorical.
Communication: the daily note thread

The Obsidian daily note has a Thread section for async conversation:

  • //c — Claude wrote this
  • //j — Jason wrote this

Bullet journal signifiers keep things scannable:

! priority
? question
~ waiting
> migrated
* idea
@ delegated
This is genuinely useful. When Jason picks up his phone and opens Obsidian, he can see what Claude did, what questions are pending, and what's blocked — without opening a terminal.
Wrap-up protocol

When Jason says "wrap up" or "close out," a full archiving sequence runs:

  1. Update the Obsidian daily note (thread, captures, close-of-day summary)
  2. Append to the filesystem log (brain-external/filesystem-log.md)
  3. Update the task board
  4. Process any Zotero items (new papers, annotations, notes)
  5. Write/update memory files for cross-session persistence
  6. Trigger the learning journal build pipeline

Nothing gets lost between sessions. That's the whole point.

$ cat --section=03-memory-system
How Claude remembers things (it's just files, all the way down)

Claude Code's memory system is elegant in its simplicity: markdown files in a known directory, indexed by a single MEMORY.md file.

Location: ~/.claude/projects/-home-jasonobrien/memory/

Memory types:

  • User — who Jason is (roles, machine, academic context, preferences)
  • Feedback — what to do and what to avoid (writing style, terminology, formatting rules)
  • Project — ongoing work (B3 research, site redesign, Zotero pipeline, vault graph)
  • Reference — where to find things (system registry, API docs, brand guidelines)
The distinction that matters: memory = cross-session persistence (survives restarts). Tasks/plans = current session only (working state). Don't put ephemeral work in memory. Don't put important context only in tasks.

When a new Claude Code session starts, it reads MEMORY.md, which indexes ~30 individual memory files. By the time Jason finishes typing his first message, Claude already knows about the B3 terminology rules, the daily note format, the TXST brand colors, and the fact that Jason is "braver than knowledgeable" about statistics.

Daily notes format

Bullet journal format in Obsidian. Each daily note has:

  • Carry-forward — tasks migrated from previous days
  • Rapid log — timestamped entries as the day happens
  • Thread — the //c / //j conversation
  • Captures — ideas, links, quotes grabbed during the day
  • Close-of-day — summary, energy level, what to carry forward
Session wrap-up checklist

The canonical sequence, every time:

  1. Daily note → thread update, captures, close-of-day
  2. Filesystem log → append file movements with keywords
  3. Task board → update statuses, add new items
  4. Zotero → process new papers, annotations, notes
  5. Memories → write/update anything that needs cross-session persistence
  6. Learning journal build → trigger the pipeline
If it didn't get wrapped up, it didn't happen. The wrap-up protocol is what makes the whole system work across sessions, devices, and context switches.

$ cat --strict b3-style-guide
Analysis conventions (violate these and the commit gets rejected)

Bobcats Bounce Back (B3) is a student success research project. The analysis pipeline has rules. They are not suggestions.

Language:

  • NEVER use causal language. It's "associated with," not "caused by." Ever.
  • B3 boosts or is associated with outcomes. It never "generates" or "creates" them.
  • Effect sizes > p-values in all reporting. Always. A tiny p-value with no practical significance is just noise with a fancy hat.

Terminology:

  • Navigator / team member — never coach, advisor, or counselor
  • Node — not hub (for campus connection points)
  • Field notes terminology follows program documentation standards

Engagement measures:

  • Contact count, intensity bands, timing patterns
  • Engagement, not eligibility (in leadership reports)
  • Per-term rates with trends, not single snapshots

Format: APA 7 throughout. No exceptions. Yes, even in the Jupyter notebooks.
